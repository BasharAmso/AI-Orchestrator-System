# docs/

This directory holds all project-specific documentation — the plans, designs, and records that describe what you're building and how.

Unlike `.claude/project/knowledge/` (which stores reusable system knowledge like decisions and glossary terms), `docs/` is where your **project deliverables** live: the PRD that defines your product, the architecture that shapes it, and the release notes that track what shipped.

---

## Suggested Starter Docs

Create these as your project evolves:

- **PRD.md** — Product Requirements Document. Describes what you're building, who it's for, core features, and success criteria. Auto-generated by the `plan-from-idea` skill when you emit an `IDEA_CAPTURED` event.

- **ARCHITECTURE.md** — High-level technical design. Covers tech stack, folder structure, data flow, and key integration points. Create this once the PRD is stable.

- **DECISIONS.md** — Project-specific design decisions that live alongside the codebase. For long-term, cross-project decisions, use `.claude/project/knowledge/DECISIONS.md` instead.

- **RELEASE_NOTES.md** — A running log of what shipped and when. Add an entry each time you complete a milestone or deploy.

- **DIAGRAMS.md** *(optional)* — Visual diagrams using Mermaid syntax (flowcharts, sequence diagrams, entity relationships). Useful for complex systems but not required for every project.

---

## When to Create Docs

| Moment | Doc to Create |
|--------|---------------|
| After your first `/run-project` with an idea | `PRD.md` |
| When choosing a tech stack or structure | `ARCHITECTURE.md` |
| When making a project-specific trade-off | `DECISIONS.md` |
| When shipping a milestone | `RELEASE_NOTES.md` |
| When a system is hard to explain in words | `DIAGRAMS.md` |

---

## Tips

- Keep docs short and scannable. Bullet points beat paragraphs.
- Update docs as the project changes — stale docs are worse than no docs.
- The AI Builder System can read these files for context, so well-written docs improve AI output quality.
